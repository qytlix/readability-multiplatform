## 1. 目标

在正式大规模实现前完成 Feed / 内容管线模块的两大基础设施：

1. **解析器选型验证** — FeedParserAdapter 封装 + 代表 Fixture 通过解析
2. **清洗链路可行性** — Readability 对代表性网页输出 Cleaned HTML

同时在 M0 阶段确认：
- 第三方库选型（rss-parser、@mozilla/readability、turndown）经原型验证
- Cleaned Content 契约初版可供 Review

## 3. 任务与拆分



### M0-FEED-01：FeedParserAdapter + Fixture（0.5～1d）

#### 范围内

- 安装 rss-parser 并验证 main 进程可加载
- 定义 `FeedParserAdapter` 接口：输入 XML/JSON 字符串 + sourceUrl，输出 `ParsedFeed`
- 实现适配器（调用 rss-parser，映射到统一模型）
- 编写测试 Fixture：
  - RSS 2.0 标准（2 个）
  - RSS 含 CDATA/中文（1 个）
  - RSS 缺字段（1 个）
  - RSS 重复 GUID（1 个）
  - Atom 标准（2 个）
  - JSON Feed 标准（2 个）
- 所有 Fixture 通过适配器解析测试
- Fixture 文件放入 `tests/fixtures/feeds/`

#### 范围外

- 同步服务（后续 M1）
- 网络获取（使用文件 fixture 即满足本任务目标）

#### 验收标准

- [ ] 适配器接口已定稿并通过类型检查
- [ ] 至少 8 个 Fixture 通过解析
- [ ] 每个 Fixture 的 ParsedFeed.title、entries 非空
- [ ] 每个 ParsedEntry 至少包含 guid、url、title
- [ ] 重复 GUID fixture 的解析结果保留最后一条
- [ ] 缺字段 fixture 不抛出异常（字段为 undefined）

---

### M0-FEED-02：正文提取可行性原型（0.5d）

#### 范围内

- 安装 `@mozilla/readability` + `jsdom` 并验证 main 进程可加载
- 使用 2～3 个代表性文章 HTML（普通文章、含表格/代码、中文文章）确认 Readability 输出
- 输出 Cleaned HTML 和清洗状态结论
- 验证 DOMPurify 或类似 sanitize 方案
- 记录可行性结论到 `docs/feed/`（或合入 M0 总结）

#### 范围外

- HTML → Markdown 转换（M1 前完成选型即可）
- 完整的 ContentCleaner 和 ContentService（M1）
- 网络获取（使用文件 fixture）

#### 验收标准

- [ ] Readability 在 main 进程可通过 jsdom 加载并运行
- [ ] 至少 2 个代表性 HTML 成功提取正文
- [ ] 输出不包含 `<script>`、事件属性、`javascript:` URL
- [ ] 记录支持边界和已知失败模式
- [ ] 如果 Readability 方案不可行，记录替代方案

## 6. 风险与停止点

| 条件 | 行动 |
|---|---|
| Readability 对代表性文章输出不可接受 | 记录边界，提供原文/Web 回退方案，不阻塞 M1 |
| rss-parser 不支持 JSON Feed 或性能不可接受 | 更换库或自行补充简单实现 |
| M0 工作量超出 2d | 优先完成 M0-FEED-01，FEED-02 可并行，超出即止 |
