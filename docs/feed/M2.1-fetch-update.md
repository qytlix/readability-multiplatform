# M2.1：ContentFetcher 三级 Fallback

> 对应全局 PLAN.md 的 M2
> 状态：**已完成**

---

## 1. 背景

### 当前问题

`ContentFetcher` 只有一级 fetch，使用固定 `User-Agent: Shale/1.0 Feed Reader`。遇到 NodeSeek (`rss.nodeseek.com`) 这类有反爬保护的论坛站点时，抓取单篇文章正文全部返回 `HTTP 403: Forbidden`。

数据库中 NodeSeek 的全部 `entry_content` 记录 `pipelineStatus = 'failed'`、`pipelineError = 'HTTP 403: Forbidden'`，无一成功。

### 适用范围

本任务只解决**文章正文内容获取**阶段的 403 问题，不影响 RSS feed 本身的同步（RSS 同步正常）。

---

## 2. 方案

三级自动降级链，各级共享同一策略接口，`ContentFetcher.fetch()` 签名不变，上游 `ContentService` 零感知。

```
Tier 0: SimpleFetch (当前现状)
   ↓ on 403/4xx/5xx
Tier 1: EnhancedFetch (UA 池轮换 + 增强请求头 + 指数退避重试)
   ↓ on 403/4xx/5xx
Tier 2: BrowserFetch  (Electron 内置 BrowserWindow 无头渲染，含 Cloudflare Challenge 处理)
```

### 3.1 FetcherStrategy 接口

```typescript
export interface FetcherStrategy {
  readonly name: string;
  fetch(url: string, signal?: AbortSignal): Promise<FetchResult>;
  isAvailable(): boolean;
}
```

### 3.2 Tier 0 — SimpleFetchStrategy

**引入库**：无

从原始 `ContentFetcher.fetch()` 提取为独立策略，行为完全相同。

- `User-Agent: Shale/1.0 Feed Reader`
- `Accept: text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8`
- 30s timeout, 10MB size limit
- HTTP 非 2xx → 抛出 `Error('HTTP {status}: {statusText}')`

### 3.3 Tier 1 — EnhancedFetchStrategy

**引入库**：无

**UA 池轮换**（5 个主流浏览器 UA，403/网络错误后自动轮换，最多 5 轮）：

```
Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 Chrome/125.0.0.0 Safari/537.36
Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:126.0) Gecko/20100101 Firefox/126.0
Mozilla/5.0 (Macintosh; Intel Mac OS X 14_5) AppleWebKit/605.1.15 Safari/604.1
Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/125.0.0.0 Safari/537.36 Edg/125.0.0.0
Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 Chrome/124.0.0.0 Safari/537.36
```

**补充请求头**：

| Header | Value |
|---|---|
| `Accept-Language` | `zh-CN,zh;q=0.9,en;q=0.8` |
| `Accept-Encoding` | `gzip, deflate, br` |
| `Cache-Control` | `max-age=0` |
| `Sec-Fetch-Dest` | `document` |
| `Sec-Fetch-Mode` | `navigate` |
| `Sec-Fetch-Site` | `none` |
| `Referer` | 从目标 URL 提取 origin |

**重试策略**：

| 条件 | 行为 |
|---|---|
| HTTP 403 / 429 / 5xx | 指数退避重试（100ms → 500ms → 2s），最多 3 次 |
| HTTP 404 / 其他非重试 | 直接抛出，不重试 |
| 网络错误（fetch rejected） | 指数退避重试（与 HTTP retry 共享重试次数） |
| AbortError | 直接抛出，不重试 |

### 3.4 Tier 2 — BrowserFetchStrategy

**引入库**：无（使用 Electron 内置 `BrowserWindow`）

**设计要点**：

1. 按需创建 `BrowserWindow({ show: false })`，用完 `destroy()`
2. 监听 `did-finish-load` / `did-fail-load`
3. **Cloudflare Challenge 自动检测**：识别 `<title>Just a moment...</title>` 或 URL 含 `__cf_chl_` 参数
4. **Challenge 等待机制**：双重保障
   - 监听后续 `did-finish-load`（重定向触发）
   - 定时轮询（2s 后每 1s 检查页面内容是否不再含 challenge 特征）
5. 返回最终页面的 URL、HTML、charset

**生命周期**：

| 阶段 | 行为 |
|---|---|
| 创建 | 每次 `fetch()` 按需创建 |
| Challenge 检测 | `<title>` 判断 + URL 参数检查 |
| Challenge 等待 | 最多 20s（Cloudflare 一般 2-5s 完成） |
| 正常页面 | 首次 `did-finish-load` 即返回 |
| 清理 | `finally` 中 `destroy()`，不留残余 |

**已知限制**：

| 限制 | 说明 |
|---|---|
| 无法获取 HTTP status code | 浏览器内部消化原始响应码，统一返回 200 |
| 无法获取原始响应 headers | 同上，返回空对象 |
| 仅在 Electron Main process 可用 | 单元测试环境 `isAvailable()` = false，自动跳过 |
| 资源开销大 | 创建销毁窗口是重量级操作，仅作为最后兜底 |

## 4. ContentFetcher 重构

```typescript
export class ContentFetcher {
  private strategies: FetcherStrategy[];

  constructor(options?: {
    maxSize?: number;
    timeoutMs?: number;
    strategies?: FetcherStrategy[];  // 允许外部注入/测试
  }) { ... }

  async fetch(url: string, signal?: AbortSignal): Promise<FetchResult> {
    let lastError: Error | null = null;
    for (const strategy of this.strategies) {
      if (!strategy.isAvailable()) continue;
      try { return await strategy.fetch(url, signal); }
      catch (err) { lastError = err as Error; }
    }
    throw lastError ?? new Error('All fetch strategies failed');
  }
}
```

向后兼容：构造函数 options 可选，`fetch()` 签名不变，原有测试全部通过。

## 5. NodeSeek 实际测试结果

### 诊断方法

对 `https://www.nodeseek.com/post-75731-1` 做了三组测试：

1. **SimpleFetch（Tier 0）**：返回 `HTTP 403 Forbidden`
2. **EnhancedFetch（Tier 1）**：同样返回 `HTTP 403 Forbidden`
3. **BrowserFetch（Tier 2）**：在 Electron Main process 中手动触发 → **成功获取渲染后 HTML**

### 403 响应分析

NodeSeek 使用 **Cloudflare JS Challenge**（`cf-mitigated: challenge`）：

```
Content-Type: text/html; charset=UTF-8
cf-mitigated: challenge
cf-ray: a1bd3e84ba1043ae-MCI
Critical-CH: Sec-CH-UA-*
```

响应体是 `<title>Just a moment...</title>` 的 Cloudflare 挑战页，包含：
- 完整 CSP 策略（仅允许 nonce-limited script 执行）
- `_cf_chl_opt` 配置对象（包含 cookie token、challenge 类型等）
- 动态加载 `challenges.cloudflare.com` 的 orchestration script

### 各层级效果对比

| 层级 | NodeSeek | 正常站点（如 GitHub Blog） | 简单反爬站点 |
|---|---|---|---|
| Tier 0 SimpleFetch | ❌ 403 | ✅ 正常 | ❌ 403 |
| Tier 1 EnhancedFetch | ❌ 403（Cloudflare 需要 JS 执行） | ✅ 正常 | ✅ 可能通过 |
| Tier 2 BrowserFetch | ✅ 通过（含 challenge 等待） | ✅ 正常 | ✅ 正常 |

### 关键发现

「增强 UA + 请求头」在 Cloudflare Challenge 面前**无效**——Cloudflare 要求：
- `Critical-CH: Sec-CH-UA-*` 等原生浏览器 Client Hints（`fetch()` 无法模拟）
- 完整的 JavaScript 执行环境来运行 challenge 脚本
- Cookie 设置和异步重定向

Tier 1 的增强头对非 Cloudflare 的简单 UA 检测（如 `curl` / `python-requests`）有效，但 Cloudflare Challenge 只能走 Tier 2。

## 9. 关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| Tier 2 实现 | Electron `BrowserWindow` 无头 | 零依赖，已有完整 Chromium |
| Cloudflare Challenge 检测 | `<title>` 判断 + URL 参数检查 | 简单可靠，不依赖外部库 |
| Challenge 等待机制 | 双重：did-finish-load 事件 + 轮询 | redirect 事件可能捕获不到 |
| 降级时机 | 策略抛异常即降级 | 每个策略内部有重试，外部单纯链式兜底 |
| BrowserWindow 生命周期 | 按需创建、用完即毁 | 不占用 GPU 和内存 |
| 是否共享 cookie/session | 不共享，每次独立 | 避免跨站 cookie 污染 |
