# M2.1：ContentFetcher 三级 Fallback

> 对应全局 PLAN.md 的 M2
> 状态：规划中

---

## 1. 背景

### 当前问题

`ContentFetcher` 只有一级 fetch，使用固定 `User-Agent: Shale/1.0 Feed Reader`。遇到 NodeSeek (`rss.nodeseek.com`) 这类有反爬保护的论坛站点时，抓取单篇文章正文全部返回 `HTTP 403: Forbidden`。

数据库中 NodeSeek 的全部 `entry_content` 记录 `pipelineStatus = 'failed'`、`pipelineError = 'HTTP 403: Forbidden'`，无一成功。

### 适用范围

本任务只解决**文章正文内容获取**阶段的 403 问题，不影响 RSS feed 本身的同步（RSS 同步正常）。

---

## 2. 方案

三级自动降级链，各级共享同一策略接口，`ContentFetcher.fetch()` 签名不变，上游 `ContentService` 零感知。

```
Tier 0: SimpleFetch (当前现状)
   ↓ on 403/4xx/5xx
Tier 1: EnhancedFetch (UA 池轮换 + 增强请求头 + 指数退避重试)
   ↓ on 403/4xx/5xx
Tier 2: BrowserFetch  (Electron 内置 BrowserWindow 无头渲染)
```

---

## 3. FetcherStrategy 接口

```typescript
// src/main/feed/FetchStrategy.ts

import type { FetchResult } from '../../shared/contracts/content.types';

export interface FetcherStrategy {
  /** 策略名称，用于日志和诊断 */
  readonly name: string;

  /**
   * 获取 URL 内容
   * @throws Error 获取失败时抛出，外部据此降级
   */
  fetch(url: string, signal?: AbortSignal): Promise<FetchResult>;

  /**
   * 检查当前环境是否支持该策略
   * Tier 2 BrowserFetchStrategy 仅在 Electron Main process 可用，此处返回 false
   */
  isAvailable(): boolean;
}
```

## 4. 三级实现

### 4.1 Tier 0 — SimpleFetchStrategy

**引入库**：无

保持当前行为不变，纯提取为独立策略。

```typescript
export class SimpleFetchStrategy implements FetcherStrategy {
  readonly name = 'simple';

  async fetch(url: string, signal?: AbortSignal): Promise<FetchResult> {
    // ── 与当前 ContentFetcher.fetch() 完全一致的实现 ──
    // UA: Shale/1.0 Feed Reader
    // Accept: text/html,application/xhtml+xml,...
    // redirect: follow
    // timeout 30s, size limit 10MB
    // HTTP 非 2xx → throw Error(`HTTP ${status}: ${statusText}`)
  }
}
```

### 4.2 Tier 1 — EnhancedFetchStrategy

**引入库**：无

使用 Node.js 原生 `fetch()`，从三方面增强请求伪装能力。

**UA 池轮换**（5 个主流浏览器 UA，403 后自动轮换）：

```
Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/125.0.0.0 Safari/537.36
Mozilla/5.0 (Windows NT 10.0; Win64; x64; rv:126.0) Gecko/20100101 Firefox/126.0
Mozilla/5.0 (Macintosh; Intel Mac OS X 14_5) AppleWebKit/605.1.15 (KHTML, like Gecko) Version/17.5 Safari/604.1
Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/125.0.0.0 Safari/537.36 Edg/125.0.0.0
Mozilla/5.0 (X11; Linux x86_64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/124.0.0.0 Safari/537.36
```

**补充请求头**：

| Header | Value |
|---|---|
| `Accept-Language` | `zh-CN,zh;q=0.9,en;q=0.8` |
| `Accept-Encoding` | `gzip, deflate, br` |
| `Cache-Control` | `max-age=0` |
| `Sec-Fetch-Dest` | `document` |
| `Sec-Fetch-Mode` | `navigate` |
| `Sec-Fetch-Site` | `none` |
| `Referer` | 从目标 URL 提取 origin，如 `https://www.nodeseek.com` |

**指数退避重试**：

| 条件 | 行为 |
|---|---|
| HTTP 403 | 轮换 UA，100ms 后重试（最多 3 次） |
| HTTP 429 | 检查 `Retry-After` 头，否则 500ms 后重试（最多 2 次） |
| 网络超时 | 直接抛出，不重试（避免雪崩） |

### 4.3 Tier 2 — BrowserFetchStrategy

**引入库**：无（使用 Electron 内置 `BrowserWindow`）

利用 Electron Main process 中可创建 `BrowserWindow` 的能力，以真正浏览器的方式加载页面。

```typescript
export class BrowserFetchStrategy implements FetcherStrategy {
  readonly name = 'browser';

  isAvailable(): boolean {
    // 仅在 Electron Main process 中可用
    return typeof process !== 'undefined' && process.type === 'browser';
  }

  async fetch(url: string, signal?: AbortSignal): Promise<FetchResult> {
    // 1. 按需创建 BrowserWindow({ show: false, ... })
    const win = new BrowserWindow({
      show: false,
      webPreferences: {
        contextIsolation: true,
        nodeIntegration: false,
        javascript: true,
      },
    });

    try {
      // 2. 加载 URL，等待 did-finish-load 或 did-fail-load
      //    设置 30s 超时（AbortSignal + timer）
      await this.loadPage(win, url, signal);

      // 3. 用 executeJavaScript 获取完整的 outerHTML
      const html = await win.webContents.executeJavaScript(
        'document.documentElement.outerHTML'
      );

      // 4. 检测字符编码
      const charset = await win.webContents.executeJavaScript(
        'document.characterSet'
      );

      return { url, statusCode: 200, headers: {}, body: html, charset };
    } finally {
      // 5. 必须 destroy，不常驻窗口
      if (!win.isDestroyed()) win.destroy();
    }
  }
}
```

**生命周期管理**：

| 阶段 | 行为 |
|---|---|
| 创建 | 每次 `fetch()` 按需创建，不常驻 |
| 加载 | `loadURL()` + `did-finish-load` / `did-fail-load` |
| 超时 | 30s timer 配合 `webContents.destroy()` |
| 清理 | 无论成功失败，finally 中调用 `destroy()` |
| 多请求 | 每次请求独立窗口，不共享 cookie/session，避免状态污染 |

**已知限制**：

| 限制 | 说明 |
|---|---|
| 无法获取 HTTP status code | Browser 内部消化了原始响应码，统一返回 200 |
| 无法获取原始响应 headers | 同上，返回空对象 |
| 仅在 Electron Main process 可用 | 单元测试环境 `isAvailable()` = false，自动跳过 |
| 资源开销大 | 每次创建销毁窗口是重量级操作，不依赖此路径处理正常请求 |

## 5. ContentFetcher 重构

### 改动要点

```typescript
// src/main/feed/ContentFetcher.ts

export class ContentFetcher {
  private strategies: FetcherStrategy[];
  private maxSize: number;
  private timeoutMs: number;

  constructor(options?: {
    maxSize?: number;
    timeoutMs?: number;
    strategies?: FetcherStrategy[];   // 允许外部注入/测试
  }) {
    this.maxSize = options?.maxSize ?? 10 * 1024 * 1024;
    this.timeoutMs = options?.timeoutMs ?? 30_000;

    // 默认策略链：Simple → Enhanced → Browser
    this.strategies = options?.strategies ?? [
      new SimpleFetchStrategy(...),
      new EnhancedFetchStrategy(...),
      new BrowserFetchStrategy(...),
    ];
  }

  async fetch(url: string, signal?: AbortSignal): Promise<FetchResult> {
    let lastError: Error | null = null;

    for (const strategy of this.strategies) {
      if (!strategy.isAvailable()) continue;

      try {
        return await strategy.fetch(url, signal);
      } catch (err) {
        lastError = err as Error;
        // 自动降级到下一级
      }
    }

    // 全部降级完毕仍失败，抛出最后一个错误
    throw lastError ?? new Error('All fetch strategies failed');
  }
}
```

### 向后兼容性

- 构造函数 `options` 可选，默认参数与当前一致
- `fetch(url, signal?)` 签名不变
- 行为变化：超时 / 5xx 抛出的异常会和原来相同的 `Error('HTTP ...')` 格式
- 只有真的遇到 403 这类"可降级错误"时才会产生额外重试请求，正常站点全程走 Tier 0，零开销

## 9. 关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| Tier 2 用什么实现 | Electron `BrowserWindow` 无头 | 零依赖，已有完整 Chromium，工程代价最小 |
| 降级时机 | 策略抛异常即降级 | 每个策略内部已有重试，外部单纯做链式兜底 |
| 是否记录每篇文章使用了哪级策略 | 可选 enhancement | JSON field 或日志中附带策略名，初期先不做 |
| BrowserWindow 生命周期 | 按需创建、用完即毁 | 不占用 GPU 和内存 |
| 是否共享 cookie/session | 不共享，每次独立 | 避免一个站点的 cookie 污染影响另一站点 |
| 降级链顺序 | Simple → Enhanced → Browser | 从轻到重，大部分站点 Tier 0 就够用 |

## 10. 验收标准

1. **Behavioral test**：注入 mock 链，T0→403 T1→403 T2→成功 → `fetch()` 返回 T2 结果
2. **Error propagation**：三级全部失败 → 抛出最后一个策略的异常
3. **Signal propagation**：AbortSignal 正确传递到各级策略
4. **Regression**：现有 content-fetcher + content-service 测试全部通过
5. **Manual test**：用 NodeSeek 的 entry 手动触发 fetch，确认能获取到非 403 的渲染 HTML
6. **Zero dependency**：无新增 npm 包
