## 1. 目标

走通 Feed/内容管线在全局"添加 Feed → 同步 → 清洗 → 持久化 → 展示"最短链路中的完整路径，让 Reader 负责人（组长）和 AI 负责人（cyj）可以基于本模块的产出物进行集成开发。

在 M1 结束时，从 Renderer 可完成：
1. 输入一个 Feed URL → 添加 Feed → 同步 → 文章列表可见
2. 点击文章 → 获取原文 → Readability 清洗 → Markdown 转换 → 结果写入数据库 → 展示纯文本或 HTML 结果
3. 重复同步不会产生重复文章

## 3. 任务与拆分

### M1-FEED-01：Feed 添加与单 Feed 手动同步（1d）

#### 范围内

**FeedService 核心流程：**

```typescript
class FeedService {
  async addFeed(url: string): Promise<{ feed: Feed; entries: EntryListItem[] }>
  async syncFeed(feedId: number): Promise<SyncResult>
  async getFeeds(): Promise<Feed[]>
  async removeFeed(feedId: number): Promise<void>
}
```

`addFeed` 流程：
1. URL 格式校验 → 失败返回 `FEED_INVALID_URL`
2. FeedStore.findByUrl() 查重 → 已存在返回 `FEED_DUPLICATE`
3. HTTP GET feedURL → 获取 XML/JSON
4. FeedParserAdapter.parse() → 解析
5. FeedStore.create() → 写入
6. EntryStore.createOrUpdate() → 写入所有文章
7. 更新 Feed.lastFetchedAt 和 syncStatus
8. 返回 Feed + EntryListItem[]

`syncFeed` 流程：
1. HTTP GET（支持 ETag / Last-Modified）
2. 304 → 无更新返回
3. 解析 → upsert Entry
4. 不同步已标记 isDeleted 的 tombstone
5. 远端 metadata 更新时不覆盖 isRead/isStarred
6. 返回新文章数

**网络层**：
- 使用 Node.js 内置 `fetch` 或 undici
- 支持 AbortSignal（预留超时机制）
- 限制响应体 5MB
- 处理重定向（最多 5 次）
- 设置 User-Agent
- 正确处理 charset（Content-Type header + HTML meta）

#### 范围外

- 并发同步（M2）
- 定时 Sync（M2）
- 完整的同步状态推送（M2）
- OPML（M2）
- 删除 Feed（M2）

#### 验收标准

- [ ] 有效 Feed URL 可添加并返回文章列表
- [ ] 无效 URL 返回 `FEED_INVALID_URL` 错误
- [ ] 重复添加返回 `FEED_DUPLICATE` 错误
- [ ] 手动同步后新文章被写入
- [ ] 重复同步不产生重复 Entry
- [ ] Feed 的 lastFetchedAt 在同步后更新
- [ ] URL 含中文/特殊字符正常处理
- [ ] 所有 Store 方法在流程中被覆盖验证

### M1-FEED-02：去重验证（0.5d）

#### 范围内

- 确保解析后的 Entry 通过 `(feedId, guid)` / `(feedId, url)` 唯一约束去重
- RSS 优先使用 `<guid>` → 回退 `<link>`
- Atom 使用 `atom:id`
- JSON Feed 使用 `id`
- 不同 Feed 可能包含相同文章（同 URL），允许共存
- 已软删除的 Entry 不因同步复活
- 使用 Fixture + 测试验证去重行为

#### 验收标准

- [ ] 同一 Feed 两次同步同一个 Entry：只保留一条记录
- [ ] 同一 Feed 更新远端 Entry 内容：UPDATE 而非 INSERT
- [ ] 不同 Feed 的同 URL 文章：两条独立记录
- [ ] tombstone（isDeleted=1）不因同步被覆盖或复活
- [ ] guid 为空的 Feed：使用 url 去重

### M1-FEED-03：单篇正文获取与清洗（1d）

#### 范围内

**ContentFetcher**：

```typescript
class ContentFetcher {
  async fetch(url: string, signal?: AbortSignal): Promise<FetchResult>
}
```

- HTTP GET + 重定向跟踪
- 大小限制 10MB
- charset 处理
- 写入 entry_content.sourceHtml 和 entry_content.sourceUrl

**ContentCleaner**：

```typescript
class ContentCleaner {
  clean(html: string, baseUrl: string): CleanResult
}
```

- 使用 @mozilla/readability + jsdom
- 输出经过 sanitize（DOMpurify 或同类）
- 提取 readabilityTitle、readabilityByline
- 写入 entry_content.cleanedHtml

**MarkdownConverter**：

```typescript
class MarkdownConverter {
  convert(html: string): string
}
```

- 安装 turndown 并配置 GFM 支持
- 写入 entry_content.cleanedMarkdown

**ContentService 整合流程**：

```typescript
class ContentService {
  async fetchAndClean(entryId: number): Promise<CleanedContent>
}
```

1. EntryStore.findById(entryId)
2. ContentFetcher.fetch(entry.url) → sourceHtml
3. ContentCleaner.clean(sourceHtml, entry.url) → cleanedHtml + title + byline
4. MarkdownConverter.convert(cleanedHtml) → cleanedMarkdown
5. ContentStore.upsert({ entryId, sourceHtml, cleanedHtml, cleanedMarkdown, pipelineStatus: 'success' })
6. 生成 sourceContentHash + segmenterVersion
7. 返回 CleanedContent

**清洗失败处理**：
1. ContentStore.upsert({ pipelineStatus: 'failed', pipelineError })
2. 返回包含 pipelineError 的 CleanedContent
3. UI 显示原文 URL 链接供用户回退

#### 验收标准

- [ ] 有效文章 URL 可获取、清洗、转换并写入数据库
- [ ] 清洗结果 cleanHtml 不包含 `<script>` 等危险标签
- [ ] cleanedMarkdown 格式为合法 Markdown
- [ ] readabilityTitle 和 readabilityByline 正确提取
- [ ] pipelineStatus 正确追踪为 'success'
- [ ] 清洗失败时 pipelineStatus = 'failed' 且 pipelineError 有值
- [ ] 已存在 content 时再次请求覆盖更新

export const contentAPI = {
