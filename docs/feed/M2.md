## 1. 目标

在 M1 最小闭环基础上，补齐 Feed / 内容管线的 P0 剩余功能，使模块达到生产可用水平。核心增量：

2. **OPML 导入/导出** — 支持嵌套 outline，merge/replace 两种模式
3. **定时 Sync** — 最小可靠版本，配置化间隔
4. **同步状态透明** — 进度推送、错误提示、重试信息
5. **清洗失败回退** — 用户可感知的降级路径
6. **Entry 星标** — 基础阅读管理

### M2-FEED-02：OPML 导入/导出（1d）

#### 范围内

**OPML 导入（ImportService）**：

```typescript
class OPMLImportService {
  async importFromFile(filePath: string, mode: 'merge' | 'replace'): Promise<OPMLImportResult>
  async importFromContent(xml: string, mode: 'merge' | 'replace'): Promise<OPMLImportResult>
}
```

- 解析嵌套 outline 结构
- `merge` 模式：新 Feed 追加，已存在跳过
- `replace` 模式：
  1. 先解析到 staging 区
  2. 新 Feed 全部创建
  3. 单事务移除不在 staging 中的旧 Feed
  4. 替换失败时自动回滚
- 合法性验证：XML 格式、根元素、基本结构
- 缺失标题的 Feed 尝试探测
- 返回结果：成功数、跳过数、失败列表

**OPML 导出（ExportService）**：

```typescript
class OPMLExportService {
  async exportToContent(): Promise<string>
  async exportToFile(filePath: string): Promise<void>
}
```

- 导出当前所有 Feed（不含已删除）为 OPML 2.0 格式
- XML 转义处理
- 先写入临时文件 → atomic rename

**IPC / Preload / UI**：

- 通道：`opml:import`、`opml:export`
- 使用 Electron `dialog.showOpenDialog` / `dialog.showSaveDialog`（文件选择由 Preload 封装）
- Renderer 按钮：导入/导出入口 + 结果反馈

#### 验收标准

- [ ] 合法 OPML 文件导入后生成对应的 Feed 和文章
- [ ] merge 模式下已有 Feed 不被重复创建
- [ ] replace 模式下旧 Feed 被移除，新 Feed 生效
- [ ] replace 失败（如 DB 错误）时旧数据保持完整
- [ ] 非法 OPML 文件显示可理解错误
- [ ] 导出文件为合法 OPML XML 格式
- [ ] 导入导出的 Feed 列表一致（幂等性验证）

### M2-FEED-03：定时 Sync 基础版（1d）

#### 范围内

```typescript
class SyncScheduler {
  start(): void
  stop(): void
  setInterval(minutes: number): void
}
```

- 应用启动时自动启动
- 按 syncIntervalMin 间隔依次同步所有 Feed
- 同一次定时触发中，Feed 之间并发，单 Feed 串行
- 使用 M2-FEED-06 的并发控制机制
- 定时器使用 `setInterval` + `clearInterval`
- 应用退出时停止定时器
- Renderer 可配置默认间隔（存于 settings 表中）

**最小调度策略**：

```
每 X 分钟：
  并发执行 syncFeed(feedId) for all active feeds
  等待全部完成（成功或失败）
  等待下一周期
```

#### 范围外

- 智能调度（错峰、按优先级、背压）— P1
- 每 Feed 独立间隔 — P1
- 定时器持久化恢复 — 停用状态下重启后不自动启动

#### 验收标准

- [ ] 启动后到达间隔自动触发同步
- [ ] 同步过程中不影响 UI 响应
- [ ] 应用退出后定时器停止
- [ ] 可配置间隔时长
- [ ] 新添加的 Feed 在下一次定时触发中被包含

### M2-FEED-04：同步状态与错误提示（0.5d）

#### 范围内

**同步进度事件推送**（`feed:sync-progress`）：

```typescript
interface FeedSyncProgress {
  feedId: number;
  feedTitle: string;
  status: 'pending' | 'fetching' | 'parsing' | 'saving' | 'done' | 'error';
  error?: string;
  newCount: number;
  totalEntries: number;
  timestamp: string;
}
```

- FeedService.syncFeed() 在关键阶段发送事件
- 全量同步时汇总进度
- 使用 IPC `send` 推送事件到 Renderer
- Preload 提供 `onSyncProgress(callback)` + 清理函数

**UI 展示**：
- Feed 列表每项显示同步状态图标（spinner / checkmark / error）
- 全量同步按钮显示状态和进度  
  ※ cherry-pick 已带来 `.sync-button.idle` / `.loading` / `.success` / `.error` 四态 CSS + spinner 旋转动画，可直接复用
- 同步通知栏或 toast

**错误持久化**：
- Feed.lastSyncStatus、lastSyncError 更新
- Renderer 可查看最后一次同步结果

#### 验收标准

- [ ] 同步过程中 Feed 项显示进行中动画
- [ ] 同步成功显示成功指示
- [ ] 同步失败显示错误标签（悬浮或点击可看详情）
- [ ] 同步进度事件在 Preload 端的 clean up 函数可用
- [ ] 重启后仍可查看上一次同步结果

### M2-FEED-05：清洗失败回退（0.5d）

#### 范围内

- ContentService.fetchAndClean 失败时明确返回 `pipelineError`
- 数据库中 pipelineStatus = 'failed' + pipelineError
- EntryDetail 组件区分：清洗成功展示内容 / 清洗失败显示原文链接
- 用户可通过原文 URL 在 Web 模式下阅读（依赖组长 Web 模式）

**错误分类与回退决策**：

| 失败类型 | 回退方案 |
|---|---|
| 网络错误（超时/断网） | 显示"网络不可用，请稍后重试"|
| Readability 清洗失败 | 显示"正文提取失败"，提供原文链接 |
| 非 HTML 内容（PDF/图片） | 不尝试清洗，直接提供原文链接 |

#### 验收标准

- [ ] 网络不可用时点击文章不崩溃，显示友好提示
- [ ] 清洗失败时提供原文 URL 链接
- [ ] 重新清洗按钮可再次触发 fetchAndClean
- [ ] pipelineStatus 和 pipelineError 正确记录

### M2-FEED-06：全量同步 + 并发控制（0.5d）

#### 范围内

```typescript
class SyncCoordinator {
  async syncAll(): Promise<SyncAllResult>
  cancelAll(): void
}
```

- 全量同步：遍历所有 active Feed 依次 syncFeed
- 并发上限默认 6（可配置 2～10）
- 同一个 Feed 不允许重复并发同步
- 单个 Feed 失败不阻断其他 Feed
- 支持整体取消（通过 AbortController）
- 记录每个 Feed 的同步结果

**IPC 变更**：
- `feed:sync` 不传 feedId → 全量同步
- `feed:sync-cancel` → 取消当前全量同步

#### 验收标准

- [ ] 全量同步遍历所有 Feed
- [ ] 并发数不超过配置上限
- [ ] 单个 Feed 失败后其他 Feed 继续同步
- [ ] 同一 Feed 重复调用 syncFeed 不会并发执行
- [ ] 取消后不再发起新请求（已发出的不可取消 — P1 完整取消）

## 6. 风险

| 风险 | 处理 |
|---|---|
| OPML replace 事务大，可能超时 | 使用 staging 表分两步提交；限制 replace 的 Feed 数量 |
| 定时 Sync 与手动 Sync 同时触发 | 加锁保护：同一 Feed 不允许重复同步 |
| 大量 Feed 同步时 UI 卡顿 | 事件批处理 + 异步更新，不阻塞 renderer 主线程 |
| 并发同步导致数据库锁竞争 | WAL 模式 + 重试机制 |
