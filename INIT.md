# Mercury 跨平台重实现项目

> 文档状态：初稿（v0.2）  
> 文档性质：项目初始需求与实施基线  
> 说明：本版本归纳当前已经明确的产品范围、计划技术栈、实现计划和初步人员分工；尚未讨论完成的细节继续保持开放。

## 1. 项目概述

本项目参考 Mercury，开发一款支持 Windows、Linux 和 macOS 的跨平台 RSS 阅读工具。

原 Mercury 已经形成从订阅源聚合、正文清洗与阅读，到大语言模型辅助处理、笔记与标签管理、文摘导出的完整产品闭环。本项目以保留和重现这一核心闭环为基础，重点解决原项目以 macOS 为主要平台所带来的平台限制，使其能够面向不同桌面操作系统的用户。

项目遵循本地优先原则，无需用户注册、登录或订阅，不主动采集用户数据。大语言模型能力保持模型中立，支持提供标准 API 的在线或本地大语言模型服务。

## 2. 目标用户

产品面向已有或潜在 RSS 阅读需求、重视阅读效率、隐私和模型选择权的跨平台桌面用户。

## 3. 项目目标

1. 在 Windows、Linux 和 macOS 上提供完整、一致的桌面阅读体验。
2. 支持 RSS 信息源的导入、同步、清洗、呈现与本地管理。
3. 将摘要、翻译和标签等大语言模型能力融入阅读流程。
4. 支持笔记、标签、筛选和文摘导出，使阅读结果能够继续整理和使用。
5. 保持本地优先，不以账号系统、项目方云端数据服务或主动遥测作为产品运行前提。
6. 保持大模型中立，支持符合标准 API 的在线模型服务和本地模型服务。
7. 在开发过程中形成持久化、有价值的 Coding Agent 工作记录和真实的团队协作记录。

## 4. 产品基本闭环

本项目当前确定的基本产品闭环如下：

1. 用户添加 Feed 或导入 OPML 订阅列表。
2. 应用同步订阅源并呈现文章列表。
3. 应用对文章内容进行清洗，并以适合阅读的 HTML、Markdown 和定制样式呈现。
4. 用户可以按需使用摘要和翻译能力辅助理解文章。
5. 用户可以通过笔记、标签和筛选整理阅读内容。
6. 用户可以将单篇或多篇阅读成果导出为文摘。

## 5. 功能需求

课程要求将功能划分为四项核心功能和四项辅助功能。四项核心功能构成主要阅读与 AI 辅助闭环；四项辅助功能用于保证产品的完整性、可维护性和阅读成果的持续使用。

### 5.1 核心功能

#### 5.1.1 Feed、OPML、Sync 与内容呈现

- 支持 Feed 解析。
- 支持 OPML 解析与订阅源导入。
- 支持订阅源同步。
- 支持文章列表与文章内容呈现。

#### 5.1.2 内容清洗与阅读呈现

- 支持生成和呈现清洗后的 HTML。
- 支持生成和呈现清洗后的 Markdown。
- 支持适合阅读的定制样式。

#### 5.1.3 Summary Agent 与 LLM Providers

- 支持使用 Summary Agent 生成文章摘要。
- 支持配置和使用不同的大语言模型 Provider。
- 支持符合标准 API 的在线或本地大语言模型服务。

#### 5.1.4 Translation Agent

- 支持使用 Translation Agent 翻译文章内容。
- 翻译能力应能够融入文章阅读流程。

### 5.2 辅助功能

#### 5.2.1 多语言、日志与调试工具

- 支持多语言界面或为多语言扩展提供基础能力。
- 提供项目运行所需的日志和调试工具。
- 日志和调试功能不得违背本地优先及不主动采集用户数据的原则。

#### 5.2.2 大语言模型用量统计

- 支持记录和呈现大语言模型的使用量。
- 用量统计服务于用户对模型使用情况的了解和管理。

#### 5.2.3 笔记与文摘导出

- 支持为文章添加笔记。
- 支持单篇文摘导出。
- 支持多篇文摘导出。

#### 5.2.4 标签系统

- 支持为文章添加标签。
- 支持按照标签筛选文章。
- 支持 Tag Agent。
- 支持标签管理。

## 6. 计划技术栈

- 桌面运行时：Electron 40 或更高版本，发布时锁定精确版本。Electron 在三个目标平台上提供一致的 Chromium 渲染环境，适合本项目大量 HTML、Markdown 和原始网页呈现需求，也便于优先验证 Windows 与原生 Wayland。
- 前端：React + 严格模式 TypeScript。React 适合构建文章列表、Reader 和多状态任务界面；TypeScript 能统一前后端类型，减少 IPC、数据模型和 AI 返回结果之间的接口错误。
- 构建与打包：Vite + Electron Forge。Vite 提供快速的开发反馈，Electron Forge 统一开发、打包和原生模块重建流程，便于生成各平台安装包。
- 后端与进程通信：Electron 主进程、Preload 与类型化 IPC；首版统一使用 TypeScript，不引入 Rust 后端或 Swift sidecar。单一语言和运行时可以降低跨平台维护成本，清晰隔离界面与数据库、网络、文件等高权限能力。
- 本地存储：SQLite + `better-sqlite3`，由主进程统一管理。SQLite 无需独立服务，符合本地优先原则；事务、迁移和结构化查询适合保存订阅、文章、任务状态及 AI 结果。
- 内容处理：Mozilla Readability + 隔离 DOM 完成正文提取，清洗并消毒 HTML，再转换为支持 GFM 的 Cleaned Markdown。Readability 适合从复杂网页中提取主要内容，能够同时为安全阅读呈现和稳定的 AI 输入提供基础。
- 阅读呈现：React 内呈现可信的清洗内容，原始网页使用隔离的 Electron `WebContentsView`。这种方式兼顾 Reader、Web、Dual 三种模式，并将不可信远程网页与应用界面隔离。
- AI 接入：主进程内实现兼容 OpenAI API 的 HTTP/SSE 客户端，共享 Provider 路由、流式响应、任务状态和用量记录。兼容标准 API 有利于接入在线及本地模型，共享基础设施可避免 Summary、Translation 和 Tag Agent 重复实现。
- 凭证管理：Electron `safeStorage`。它能够调用各平台的系统级凭证保护能力，避免以明文保存 API Key。
- 测试与发布：Vitest、React Testing Library、Playwright；使用 GitHub Actions 在各目标系统的原生 Runner 上构建与验证。该组合覆盖逻辑、界面和完整用户流程，并能发现原生模块、安装包及平台行为差异。

整体采用“React Renderer → 类型化 Preload IPC → Electron 主进程”的分层结构。Renderer 负责界面与交互状态，主进程负责数据库、Feed 同步、内容清洗、AI 任务、文件和凭证；IPC 边界进行运行时校验。

## 7. 简单实现计划

1. **垂直切片**：搭建 Electron、React、TypeScript、Vite 与 Forge 工程，建立安全窗口、IPC 和数据库基础；跑通添加 Feed/导入 OPML、同步、文章列表及单篇文章清洗，并在 Windows 11 与原生 Wayland 环境验证。
2. **阅读基础**：完成 Feed 管理与定时同步，以及 Reader、Web、Dual 模式、主题、多语言、已读状态、搜索和离线能力。
3. **AI 基础**：完成 Provider/模型配置、API Key 安全存储、共享流式任务框架、Summary、Translation 和用量记录。
4. **辅助功能**：完成笔记与单篇/多篇文摘导出、手动与 AI 标签、批量标签、标签管理、日志与诊断导出。
5. **发布加固**：补齐单元、集成和端到端测试，验证 Windows、Linux 原生 Wayland/XWayland 与 macOS 打包流程，并通过 GitHub Actions 产出发布构建。

## 8. 人员分工

- **陈一诺**：公共 IPC、需求与整体交互梳理；前端基础框架和核心页面；Reader 和清洗结果展示；完整 Summary 功能链路。
- **秦一天**：内容清洗相关后端，Feed/OPML/Sync 后端,包括从原始内容到 Cleaned HTML、Cleaned Markdown 的处理及其接口。
- **陈彦杰**：AI 相关后端，重点负责 Translation 等功能；参与前端细节优化和后端运行效率优化。
- **后续待定**：数据库与任务框架的详细归属；四项辅助功能的具体分工。

成员在各自负责链路中共同遵守统一的 IPC、数据库、任务状态、错误处理与测试契约，跨模块接口通过 Issue 或设计记录确认。
注：公共 IPC 只负责通信底座和规范；每项功能的实现者负责定义并实现自己的具体 IPC，但必须遵守公共规范。

## 9. 非功能需求与技术约束

### 9.1 产品体验

- 遵循良好的设计规范。
- 提供优雅、简洁的用户体验。
- 核心功能与辅助功能应共同构成完整产品。

### 9.2 本地优先

- 无需注册、登录或订阅即可使用产品。
- 产品不主动采集用户数据。
- 产品设计和后续技术实现应以本地数据与用户控制权为基础。

### 9.3 平台中立

- 支持 Windows。
- 支持 Linux。
- 支持 macOS。

### 9.4 大模型中立

- 不将产品绑定到单一模型或单一模型服务商。
- 支持任何提供标准 API 的大语言模型服务。
- 支持本地大语言模型服务。

### 9.5 Coding Agent 留痕

- Coding Agent 的工作过程需要形成持久化、有价值的文档。
- 工作记录应服务于需求追踪、方案理解、实现审查和后续维护，而不只是保存未经整理的对话内容。

### 9.6 团队协作留痕

- 正确记录每次提交的实际提交人。
- 保留真实、清晰的提交历史。
- 后续开发过程应通过可追踪的任务、代码修改、审查和验收记录体现团队协作。

## 10. 当前范围与边界

本项目当前已经明确的边界如下：

1. 产品形态为面向 Windows、Linux 和 macOS 用户的跨平台桌面工具。
2. 产品以 RSS 阅读、AI 辅助理解和阅读成果整理为核心。
3. 产品遵循本地优先原则，不以项目方账号系统或主动采集用户数据为运行前提。
4. 产品不绑定单一大语言模型或服务商。
5. 四项核心功能和四项辅助功能均属于课程给定的产品范围。
6. 当前对功能的分层表示开发与体验层级，不表示任意一组课程要求可以直接删除。
7. 首版采用单一 Electron/TypeScript 运行时，不引入 Rust 后端、Swift sidecar 或第二套应用运行时。
